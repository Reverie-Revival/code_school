**How to apply:** When James wants to work on something, check here first for context before starting. When a new idea comes up mid-session, add it here rather than just fixing it inline (unless it's small enough to just do). When something is built, move it to **Built, awaiting James's QA** — do NOT remove it yet, even if it's been tested end-to-end in an automated browser test. Only remove an item once James has personally tried it himself and confirmed it's good. Automated verification proves it works; it doesn't substitute for James's own sign-off before something leaves the list.

**Every time one or more items move into "Built, awaiting James's QA," give James an explicit checklist of what to click and what he should see** — not just "done, go check it out." ## Open ideas

1. **Practice-check strictness needs some grace, but not uniformly.** James hit this in real usage: told to type `Hello, Python!` exactly, missed something small, and got blocked — because lesson-locking means a failed check genuinely blocks progress. But blanket leniency isn't right either: Lesson 1.2's whole point is "capitalization and punctuation matter to Python," so exact-match *is* the lesson there, not a bug. Partially eased by the new Help/Reset buttons (2026-07-19, see Built section) — a stuck kid isn't blocked forever now — but the underlying "should some lessons accept near-misses" design question is still open.

2. **Badge system for profiles**, showing what a kid has completed. Explicitly future/low-priority — James flagged it just so it isn't lost, not because it's needed now (added 2026-07-18).

## Built, awaiting James's QA

1. **Sticky lesson-pane header** (2026-07-20): Prev/Next (and each pane's own header) now stay pinned to the top of their pane while its content scrolls underneath, instead of scrolling away with the text.
   - Open a lesson with enough content to scroll (e.g. 2.2 or 2.3), scroll down, confirm Prev/Next stay visible at the top instead of disappearing.

2. **Help/Reset now show up on errors, not just wrong output** (2026-07-20): previously, if a practice attempt errored out (e.g. a typo'd variable name) there was no output at all, so the Help/Reset buttons — which lived only in the Output pane — never appeared, leaving a stuck kid with no way out. They now also appear inside the red "How To Fix It" box whenever the run errors.
   - On a lesson with a known-solution practice (e.g. 1.2 or 2.2), type something that errors and press Run. Confirm Help/Reset appear in the Errors panel, and Help fills in working code.

3. **Project content type — mechanism** (2026-07-19). `chapter.project` is a real, renderable shape (`js/app.js`'s `renderProject()`) with its own dropdown entry, a "When You're Done" box, and an **I showed someone! ✓** button. This is the **self check-in** design, not a parent gate: the button stays disabled until the kid's code runs with no error (reusing the Errors panel), and clicking it — no parent action needed — is what unlocks the next chapter. A `progress` row records it (`lesson_number: null`, `completed_at` set). CLAUDE.md's "Parent gate" principle is updated to match (now called "Self check-in"); the now-unused `parent_approved_at` column was dropped from the schema (2026-07-20 migration) since James was emphatic ("I HATE THE PARENT GATE") that it should be gone everywhere, not just unused. First build of this (mid-session) actually implemented a real parent-approval gate per CLAUDE.md's *then*-current wording — caught before handoff that this directly contradicted James's explicit 2026-07-18 feedback ("does NOT want a parent-gate blocking chapter-to-chapter progression"), asked James directly, and he confirmed self check-in. Chapter 3's Mad Libs (item 4 below) is the first chapter to actually exercise this mechanism with real content — test them together.

5. **input() actually works now** (2026-07-20, refined same session after James's own testing caught two follow-on issues). `input()` previously threw `OSError: I/O error` immediately (no stdin was ever wired up). The current design: `js/app.js`'s `runCurrentCode()` shadows the `input` builtin with a small Python wrapper (`_browser_input`) that calls a JS bridge (`_browser_prompt`, set up once in `startApp()`) to show a real browser `prompt()` popup — the question text you write in `input("...")` appears **in the popup itself**, not in the Output pane, and the Output pane shows only what your code actually `print()`s. If the popup gets closed/cancelled without an answer, Python raises `EOFError`, which `js/errorHints.js` now translates into a specific plain-English hint ("looks like the input box was closed... press Run again and type an answer").
   - James's testing also caught that **input()'s prompt text used to leak into the same output the practice checks inspect** (before the popup-based redesign above), which meant every check in **5.2, 5.3, 5.4, and 7.3** could never turn green for anyone. Those four checks were also loosened to match the meaningful part of the output rather than anchoring to its very start — belt-and-suspenders with the popup fix, in case output ever picks up an unexpected prefix again. Re-verified all four both pass on a correct answer and still correctly fail on a wrong one.
   - **Please specifically re-test 5.2, 5.3, 5.4, and 7.3** — these were broken for every kid, not an edge case, so they're the highest-priority re-check in this whole batch. Confirm: the question shows in the popup (not the Output pane), the Output pane shows only clean print() results, and cancelling the popup shows the new EOFError hint in the Errors panel.

6. **Chapters 4–15 — the entire rest of the curriculum** (2026-07-20): every remaining chapter is now written and wired into `curriculum/python/chapters/index.js`, finishing the full 15-chapter course end to end. This is a lot to review — a few notes to make it faster:
   - **Chapter 13 (Functions II) has no Project by design** — its calculator checkpoint is folded directly into lesson 13.3's practice, per the curriculum outline. If the dropdown shows no Project entry for Chapter 13, that's expected, not missing content.
   - **Chapters 6, 12.2, and 15.2 have practice checks that accept more than one exact answer** — 6.4 has no auto-check (open creative, like earlier themed wrap-ups), while 12.2 (rolls a `random.randint`) and 15.2 (rolls a die) accept whichever of the two valid outcomes actually happened, since the correct answer is genuinely random each run. If a check ever looks like it passed "too easily," that's most likely why, not a bug.
   - **The Chapter 15 capstone (Dice Dungeon)** combines nearly everything from the whole course — dictionaries, lists, loops, functions, randomness, input, decisions. Run it a few times; since it's randomized, try enough runs to see both a survival ending ("You made it through the dungeon!") and a defeat ending ("Better luck next time!") — an early bug where both endings printed back-to-back on a defeat was caught and fixed before this handoff.
   - Beyond those flagged spots, the fastest way to review is chapter by chapter: work each lesson's Your Turn, confirm the Example's shape matches it, and open each Project once its chapter's lessons are done (Math Magic Trick, Pig Latin Translator, Choose Your Own Adventure, Crack the Code, Pattern Art Machine, Dice Dungeon).
