Keep it lean: docs only where they'll actually get read again (docs/architecture, docs/decisions), no speculative scaffolding for folders/tooling not yet in use. This is a solo family project — don't over-formalize process for its own sake.

**Why:** Default Reverie Revival preference, doubly true here since James is the only builder and the audience (his kids) cares about the app, not the docs.

**How to apply:** Prefer a short ADR in docs/decisions/ over a long design doc; skip docs/specs or docs/learning unless a real need shows up (e.g. James wants to explain a concept to the kids in writing).
